# Chirp API — Refactoring Audit

## Issue 1: The Credential Problem

### Vulnerability 1A: Insecure Password Hashing (Critical)

**File:** `apps/api/chirp_api/services/utils.py`

**What it was:**
```python
def hash_password(password: str) -> str:
    return hashlib.sha256((password + "salt").encode()).hexdigest()
```

**Why it's critical:**
- SHA-256 is a general-purpose hash, not a password KDF. A modern GPU can compute billions of SHA-256 hashes per second, making offline brute-force trivial.
- The salt is hardcoded as the string `"salt"` — identical for every user. This means:
  - Identical passwords produce identical hashes (rainbow table attack is viable).
  - An attacker with the database can crack all passwords in one batch pass.
- No cost factor: the computation time cannot be tuned upward as hardware improves.

**How it could be exploited:**
An attacker who gains read access to the database (SQL injection, backup leak, insider threat) can download all password hashes and crack them offline using precomputed rainbow tables or a GPU brute-force rig. With SHA-256 + static salt, cracking `password123` takes seconds.

**Fix:** bcrypt with `rounds=12` and a per-password random salt generated by `bcrypt.gensalt()`. bcrypt is ~100ms per attempt by design, making brute force infeasible at scale.

**Migration strategy (backwards-compatible):**
- Legacy hashes are 64-char hex strings. Bcrypt hashes start with `$2b$`.
- `verify_password()` detects the format and uses the correct verification path.
- On the first successful login with a legacy hash, `login_user()` immediately rehashes with bcrypt and persists the new hash. Users migrate transparently without a forced reset or knowledge of their plaintext password.
- New registrations always use bcrypt.

### Vulnerability 1B: Hardcoded JWT Secret (Critical)

**File:** `apps/api/chirp_api/middleware/auth.py`

**What it was:**
```python
JWT_SECRET = os.environ.get("GRPC_JWT_SECRET", "chirp-grpc-jwt-secret-key-at-least-32-chars")
```

**Why it's critical:**
Anyone with access to the source code (developers, contractors, attackers with code access) knows the fallback secret. They can forge a JWT for any user ID, including admins, with no database access required — a complete authentication bypass.

**Fix:** If `GRPC_JWT_SECRET` is not set, the server raises `RuntimeError` at import time and refuses to start. A secret must be explicitly provisioned. Tests set `GRPC_JWT_SECRET` in `conftest.py` before any imports.

## Issue 2: The Query Performance Problem

### Anti-pattern: N+1 Queries

**Files:** `feed_service.py`, `posts_service.py`, `bookmarks_service.py`

**Pattern identified:**
Each service fetched a list of N posts, then looped over them calling `_get_post_counts()` — which itself issued 2–3 scalar subqueries per post. For 10 posts:

| Operation | Before | After |
|-----------|--------|-------|
| Home feed (10 posts) | 31 queries | 3 queries |
| User profile posts (N posts) | 3N+1 queries | 3 queries |
| Bookmarks page (10 posts) | 41 queries | 4 queries |

**Root cause:** Fetching aggregates inside a loop instead of using a single `GROUP BY` query over the full result set.

**Fix:** `apps/api/chirp_api/services/query_helpers.py` — `bulk_get_post_stats()` issues two aggregate queries (one for likes, one for comments) over the full set of post IDs, then a single filtered query for viewer like status. All list operations now call this instead of looping. Single-post lookups retain targeted queries (N=1, no change needed).

**Reusable pattern:** Any new feature that displays a list of posts must use `bulk_get_post_stats()` + `attach_post_stats()` rather than per-row counts. This is documented in `query_helpers.py`.

## Issue 3: Error Handling & Observability

### Problems identified:

1. **No request tracing.** Log lines had no correlation ID, making it impossible to reconstruct the sequence of events for a single failing request in production.

2. **Inconsistent error strategies across handlers:**
   - `auth_handler.GetCurrentUser`: `context.abort(UNAUTHENTICATED)` — correct gRPC idiom
   - `posts_handler.GetPost`: returns empty `PostResponse` on error — silent failure, client can't distinguish "not found" from "server error"
   - `feed_handler.GetHomeFeed`: no try/except — raw Python exception propagates as gRPC UNKNOWN
   - `admin_handler.ListUsers`: no try/except — same
   - `posts_handler.CreatePost`: `success=False` with error string in proto — uses application-level error, not gRPC status

3. **No structured logging.** Log lines lacked method name, user ID, and timing context.

### Fix: `apps/api/chirp_api/middleware/tracing.py`

- `request_trace(method_name, user_id)`: context manager that generates a UUID trace ID, stores it in a `ContextVar` (thread/coroutine safe), and logs start + completion with elapsed time.
- `get_trace_id()`: returns the current trace ID from any code in the call stack.
- `classify_error(exception)`: maps exception message patterns to gRPC status codes using a priority-ordered lookup table.
- `handle_grpc_error(error, context, method_name)`: logs the error with trace ID and calls `context.abort()` with the correct status code.

## Issue 4: Test Infrastructure

### Problems identified:

1. **Global counter state in `helpers.py`:** `_counter` is a module-level global that increments across tests. If tests run in non-deterministic order or in parallel, counter values bleed between tests. Fixed by using `generate_id()` directly for uniqueness instead of a sequential counter.

2. **Missing test coverage:**
   - No tests for `bookmarks_service`, `feed_service`, `search_service`, `notifications_service`, `mentions_service`
   - No tests for error paths in handlers (what happens when the service raises)
   - No query count regression tests

3. **Handler tests patch at the wrong level:** Some handler tests patch `SessionLocal` at the module level rather than using the `session` fixture. This works but is fragile — the mock context manager setup is repeated verbatim across every test.

New tests added:
- `tests/services/test_credential_security.py` — 15 tests for Issue 1
- `tests/services/test_query_performance.py` — 6 tests for Issue 2 with query counters

```bash
# Install pre-commit hook
cat > .git/hooks/pre-commit << 'HOOK'
#!/bin/sh
cd apps/api
GRPC_JWT_SECRET=test-secret uv run --with pytest pytest tests/ -q --tb=short
HOOK
chmod +x .git/hooks/pre-commit
```
